## 라이브가 계약 테스트를 이겼다

계약 테스트 15개가 전부 통과한 상태에서 첫 실호출이 **즉시 실패**했다.

```
CodexImageError: codex 실행 실패: spawn codex ENOENT
```

두 가지가 겹쳤다. ① Windows에서 `codex`는 npm 전역 설치의 **`.cmd` 셸 심**이라 `shell` 없이 `spawn`하면 못 찾는다. ② 지시문을 **argv에 실었다** — 줄바꿈·따옴표를 품은 문자열이라 셸 경유가 필요해지는 순간 인용 규칙에 걸린다.

**계약 테스트는 이걸 잡을 수 없었다.** 실행기를 주입해 대체하는 게 그 테스트의 설계이므로, 실행기 자체는 라이브에서만 검증된다. 주입 가능한 설계는 재현성을 위해 옳았지만, 그 대가로 실행기 하나가 검증 사각지대에 남는다는 걸 라이브가 알려줬다.

수정: 지시문을 **stdin**으로 넘긴다(`codex exec -`). argv에는 고정 플래그만 남으므로 Windows에서 `shell: true`를 켜도 주입 위험이 없다.

## 계약 셋

1. **생성물이 정말 PNG인지 파일 헤더로 확인한다** — 시그니처 8바이트 + 첫 청크 IHDR. (출처: PNG Specification, W3C — https://www.w3.org/TR/png/ , 접근 2026-07-20)
2. **실제 치수를 기록한다** — 요청 치수가 아니라 파일이 말하는 값. 구 어댑터가 요청 치수를 박아넣던 것이 실패 원인 중 하나였다.
3. **cover 잘림이 1/3을 넘으면 거부한다** — 종횡비 차이 대신 **잘림 면적 비율**을 임계로 삼았다. "종횡비 차이 0.2"는 판단이 안 되지만 "원본의 32%가 잘린다"는 바로 판단된다.

## 치수 정확성 — 단정하지 않는다

| 표본 | 요청 | 반환 |
|---|---|---|
| 사전 probe (2026-07-20) | 정사각 | 1254×1254 (16의 배수 아님) |
| 이번 실호출 | 1000×800 | **1000×800 (일치)** |

표본 2개로는 규칙을 알 수 없다. 그래서 공급자는 여전히 **파일 헤더를 정본**으로 읽는다 — 요청 치수를 믿는 구현으로 돌아가지 않는다.

## finding 큐

- `shell: true`로 Node `DEP0190` 경고가 남는다. argv는 고정 리터럴뿐이라 주입 위험은 없다 — `codex.cmd` 절대경로 해석으로 없앨 수 있다.
- 소재 URI가 파일 경로다(data URI 아님). 문서가 파일 시스템에 의존하므로 왕복·배포 시 경로 규약이 필요하다.
- fixture PNG가 1.5MB다. 소재가 늘면 레포 크기 관리 필요.
- PNG만 받는다. Codex가 webp·jpeg를 낼 수 있는지 미확인.
- 공급자 계약 검증(mime·크기)이 이제 실재한다 — TH4에서 뺐던 `provider-invalid-mime-size` 거부 경로를 verify 매니페스트에 되살릴 수 있다.
