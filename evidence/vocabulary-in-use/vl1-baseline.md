## 2. 계획 대비 편차 — 군집 15개 → 12개

plan의 verify는 "군집 15개 전부 ≥2건"이었다. **실제로는 12군집 32건이다.** 이유를 그대로 적는다.

빠진 3군집은 **정답 라벨을 댈 외부 원문을 찾지 못했다**:

| 빠진 군집 | 상태 |
|---|---|
| 내비게이션 범위 (breadcrumb / section tabs / sidebar) | breadcrumb의 *부정* 조건("flat hierarchies 1~2단계면 불필요")은 확보했으나, "이 상황이면 이걸 쓰라"는 **긍정 라벨**을 못 찾았다. 부정문에서 긍정 답을 추론해 라벨링하면 그 순간 출처가 아니라 우리 해석이 정답이 된다 |
| 많은 선택지 (combobox / select / radio) | 옵션 수백~수천에서 검색형 콤보박스를 쓰라는 서술을 원문으로 확인 실패 |
| 카드형 펼침 (disclosure-card / accordion / 더보기) | 확인 실패 |
| (부분) 필터 패널 | 찾기 군집에 포함하려 했으나 "언제 필터 패널을 쓰라"는 원문 확인 실패 — search field / command palette 2건만 남겼다 |

**지어내지 않았다.** 그럴듯한 통념 한 줄이 측정 전체를 무효로 만든다. 이 3군집은 VL5에서 군집 데이터를 쓸 때 원문을 다시 찾고, 확보되면 케이스를 **추가**한다(기존 32건은 고정 — 자를 바꾸면 대조가 무너진다).

또한 근거가 약한 인용 2건(`tabs-used-right`, `checkboxes-design-guidelines` — fetch 요약 모델을 거쳐 원문 1:1 대조를 못 함)은 **라벨에 쓰지 않았다**. 같은 군집을 다른 확인된 원문으로 채웠다.

## 3. 측정 조건

- **판정자**: 격리된 백그라운드 서브에이전트(sonnet). 부모는 이미 갭을 다 읽은 상태라 기준선을 오염시킨다.
- **제공 자산**: 현행 배포본만 — `llms.txt` → `entry-protocol` → 그 문서가 지시하는 자산(taxonomy·recipes). 용어 사전도, 결정 데이터도 없다(둘 다 아직 존재하지 않는다).
- **금지**: 외부 UX 문헌(NN/G·uxpatterns·Apple HIG·Material) 검색·fetch 금지. 정답 라벨의 출처를 판정자가 직접 읽으면 측정이 아니라 답안 열람이 된다.
- **채점**: `node scripts/score-decision-cases.mjs <answers.json>`. 자기채점 금지 — 판정자는 답과 근거만 낸다.
- **표기 정규화**: 대소문자·공백·하이픈 차이는 같은 답으로 본다. 측정 대상이 표기법이 아니기 때문이다.

### 표기 동의어 표를 나중에 넣은 것에 대해

`ALIASES`(scorer)는 기준선 답을 **본 뒤에** 추가했다. 자를 사후에 손댄 것이므로 밝혀둔다:

- 넣은 것은 **판정이 아니라 표기만 다른 쌍**뿐이다. 의미가 갈리는 쌍(예: `toast`↔`popover`)은 넣지 않았다.
- 방향이 보수적이다 — 기준선을 65.6%에서 84.4%로 **올렸다**. 처치(VL7)가 넘어야 할 문턱을 높이는 쪽이지, 향상 폭을 부풀리는 쪽이 아니다.
- 기준선과 처치가 **같은 표**를 쓰므로 대조는 그대로 성립한다. VL7은 이 표를 바꾸지 않는다.

### 오답 5건의 성격

| 케이스 | 답 | 라벨 | 성격 |
|---|---|---|---|
| overlay-3 | modal | drawer | **진짜 오판** — 스크롤이 필요하면 side-panel/full-page로 올리라는 Carbon 규칙을 모른다 |
| single-choice-3 | segmented control | radio | **진짜 오판** — 공간이 있으면 라디오로 전부 노출하라는 규칙을 모른다 |
| waiting-3 | skeleton | spinner | **진짜 오판** — 단일 모듈이면 스피너라는 구분을 모른다 |
| floating-1 | toast | popover | **케이스 결함** — 요구 문장("뒤 화면 계속 조작 가능·급하지 않음")을 토스트도 만족한다. 내가 쓴 문장이 답을 하나로 좁히지 못했다 |
| floating-2 | dropdown menu | popover | **자산 대 라벨 충돌** — 판정자가 우리 레시피(`popover.md`)의 "단순 명령 목록이면 dropdown-menu"를 근거로 댔다. Polaris 라벨과 충돌하며, 이 건은 우리 자산 쪽이 더 정확할 수 있다 |

결함 2건을 **고치지 않고 남긴다.** 답을 본 뒤 케이스를 손보면 그때부터 자가 아니라 답안이다. VL7도 같은 32건으로 채점하므로 대조는 유효하고, 이 2건은 양쪽 모두에서 동일하게 작동한다.

## 5. 판정 — 이 수치가 horizon 전제에 던지는 것

**기준선 84.4%는 예상보다 훨씬 높다.** 그리고 12군집 중 **8군집이 이미 만점**이다(펼침류·온오프·목록·더 불러오기·찾기·알림·긴 폼·모바일 전환).

이 계측이 말하는 것:

1. **결정 계층의 향상 여지는 32건 중 5건, 그중 진짜 오판은 3건이다.** VL4·VL5가 군집 15개에 판별 축을 채우는 계획이었는데, 그중 8군집은 **애초에 안 헷갈린다**는 것이 수치로 나왔다. VL5의 군집 선정은 이 장부를 근거로 좁혀야 한다 — 만점 군집에 축을 다는 것은 비용만 든다.
2. **틀린 3건의 공통점**: 전부 *임계값이 있는 규칙*이다(스크롤이 필요할 만큼 길면 / 공간이 있으면 / 단일 모듈이면). 모델이 일반 상식으로 아는 것과 못 아는 것의 경계가 여기다 — 판별 축의 값어치는 "어떤 요소가 있는지"가 아니라 **"어디서 갈리는지"**에 있다.
3. **떠있는 것 군집 0/2는 두 건 다 측정 결함**이다(케이스 모호 + 자산 대 라벨 충돌). 이 군집은 기준선을 신뢰할 수 없다.
4. **VL2·VL3(어휘 배포·참조 복구)의 근거는 흔들리지 않는다.** 이 측정은 "요소를 고를 수 있는가"만 재고, "고른 요소를 만들 수 있는가"는 재지 않는다. 481개 용어에 자산이 없다는 사실과 끊긴 참조 91건은 그대로다.

**사용자 판단이 필요한 지점**: VL4·VL5·VL7(결정 계층)의 규모를 이 수치에 맞춰 줄일지 여부. VL1의 소관은 재는 것까지이므로 여기서는 기록만 하고, milestone 보고에서 제안한다.
