# 진입 규약이 매체를 먼저 정한다

> 2026-07-22 · milestone DOG6 step-1 · changeset #264

## 왜

DOG4가 매체별 게이트를 문서로 세웠고 DOG5가 슬라이드를 채웠다. 그런데 **에이전트가 실제로 따르는 절차 문서**(`entry-protocol.md`)는 여전히 화면 하나만 알았다 — step 4가 "상태·다크모드·대비"였다.

분류표가 있어도 절차가 안 물으면 아무 일도 안 일어난다.

## 무엇을

**`N-0. Pin the medium`** — 매체 판정을 `Before the task branches`의 **첫 단계**로 넣었다. 질문 하나: *"이 산출물은 상호작용하는가?"* 누르고 입력하고 이동하면 화면, 읽기만 하면 지면, 넘겨가며 말하면 발표.

**step 4를 매체별로 갈랐다** — 세 가지가 서로의 기준을 베끼지 않는다:

| 매체 | 게이트 |
|---|---|
| 화면 | 상호작용 상태 · 다크모드 · WCAG · 반응형 경계 |
| 인쇄 | 도련·안전영역(`validatePrintSpec`) · **지면 기본은 흰 종이** · **최종 형식 래스터화**(PDF는 PDF를 렌더, 첫·중간·끝 3면) |
| 발표 | 프리셋 선언 대조(`validateSlideDeclaration`) · WCAG AA · **통설은 옵트인 경고이지 통과 조건이 아님** |

## 왜 끝이 아니라 맨 앞인가

**매체는 무엇을 검사할지가 아니라 무엇을 만들지를 바꾼다.** 종이에는 hover 상태도 다크모드도 없고, 슬라이드에는 반응형 브레이크포인트가 없다. 다 만들고 나서 매체를 정하면 다시 만들어야 한다.

그리고 한 프로젝트가 여러 매체를 낳으면 **산출물마다 자기 게이트를 돈다.** 화면 게이트를 통과한 컴포넌트는 그것이 인쇄되는 지면에 대해서는 아무것도 통과하지 않았다 — 어록 PDF에서 일어난 일이 정확히 그것이다.

## 규칙 본문을 복제하지 않았다

게이트의 구체 기준은 각 매체 정본 문서(`print-spec.md`·`slide-spec.md`)를 **링크로** 가리킨다. entry-protocol은 "어느 자를 대는가"만 말하고 자의 눈금은 갖지 않는다.

## Failure probe

매체 판정 단계와 인쇄·발표 게이트 줄을 지우면 → **5건 실패**(세 게이트 줄 누락, 래스터화·도련·프리셋 대조 부재, 판정 단계 부재). 원복 후 통과.

이 상태가 되면 인쇄·슬라이드 작업에서도 "라이트/다크 스크린샷"만 지시하게 된다 — horizon 닫는 기준 6이 관측 불가능해지는 지점이다.

## Contract

- source of truth: `docs/design-system/entry-protocol.md` (매체 분기 규칙) — 게이트 눈금은 각 매체 문서
- deploy/sync target: `llms.txt` → Cloudflare Pages
- out of scope: 검사기 호출 배선(step-2), 자가 수정 루프 실증(step-3)
