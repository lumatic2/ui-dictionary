## 무엇을 바꿨나

- **`text-fitting.ts` 신설** — 문자 종류별 전진폭 근사(`estimateWidth`), 어절 단위 줄 나눔(`wrapLines`), 상자에 맞는 최대 크기 탐색(`fitText`), 넘침 측정(`measureOverflow`).
- **`TextFitError`** — 최소 크기(14px)로도 안 들어가면 **던진다**. 잘린 채 내보내지 않는다.
- **실측 fixture** — `__fixtures__/browser-text-metrics.json`에 브라우저 `canvas.measureText` 실측 60건(폰트 2 × 크기 3 × 표본 10)을 저장했다.

## 모델은 근사다 — 그래서 오차를 쟀다

실제 폰트 파일 메트릭을 읽지 않고 문자 종류별 평균 전진폭으로 추정한다. 근사를 정확한 척하지 않기 위해 브라우저 실측과 대조했다.

**오차는 비대칭으로 다룬다.**

- **과소추정(모델 < 실제)은 위험하다** — 안 들어가는 글자를 들어간다고 판단해 잘린 산출물이 나간다.
- **과대추정은 안전하다** — 글자가 필요보다 작아질 뿐이다.

그래서 허용치를 다르게 뒀다: 과소 **8%**, 과대 **25%**. 테스트가 실측 60건 각각에 대해 이 경계를 단언한다.

### 보정 전후

첫 계수로는 과소추정이 최대 **-19.8%**였다(`punct`, `latin-lower`). 이건 위험한 방향이라 계수를 올려 재측정했다.

| 표본 | 보정 전 | 보정 후 |
|---|---|---|
| punct / sans | **-19.8%** | -3.3% |
| latin-lower / sans | **-19.2%** | -6.4% |
| latin-caps / serif | -9.0% | -6.0% |
| hangul-long / sans | +9.0% | +9.6% |
| thin / serif | -0.4% | +17.4% |

보정 후 **가장 큰 과소추정은 -6.4%**이고 나머지는 대부분 과대(안전) 방향이다. `thin/serif`의 +17.4%는 과대라 안전하지만 글자가 불필요하게 작아지는 대가가 있다 — 허용치 25% 안이다.

### Failure probe — 허용치가 실제로 막는가

계수 보정 **전** 상태에서 테스트를 돌리면 `punct`·`latin-lower` 케이스가 과소추정 8% 경계에 걸려 실패한다. 이게 이 게이트의 존재 이유다 — 허용치를 사후에 늘려 통과시키지 않았고, 계수를 고쳐 통과시켰다.

들어가지 않는 텍스트 probe: 60×20 상자에 긴 한국어 문장 → `TEXT_DOES_NOT_FIT`, 메시지에 최소 크기 포함.

### 넘침 정의를 한 번 고쳤다

첫 `overflowAmount`는 줄바꿈만 보고 폭 초과만 쟀다. 그러자 알려진 넘침 2건이 **0으로 나왔다** — 긴 문장이 여러 줄로 접히며 폭 초과는 사라지지만 상자 **높이**를 넘어 잘리기 때문이다. 그게 TH7이 산출물에서 본 실제 모습이다. 폭과 높이를 둘 다 보도록 `measureOverflow`로 바꿔 두 건을 다시 잡았다.

## finding 큐

- 실측 표본이 Windows Chrome 1종 환경이다. 다른 OS·폰트 폴백에서는 값이 다를 수 있다 — 인쇄 발주 전에 실제 조판 환경에서 재측정할 대상.
