인쇄 규격을 선언하고(step-1) 프리셋을 갖췄지만(step-2), **내보낸 산출물에는 도련도 재단 표시도 없었다.** 재단 크기 그대로 나가면 인쇄소가 받을 수 없다.

## 왜 SVG에 직접 그리는가

CSS `@page`의 `bleed`·`marks`는 **어느 브라우저에도 지원이 없다**(리서치 3.1~3.3). Chrome GUI 인쇄는 `size`조차 무시하고, headless는 `size`만 반영한다. Prince(상용)만 완전 지원, WeasyPrint는 도련 미지원, Paged.js는 폴리필 수준이다.

사용자 확정(2026-07-20)대로 **벡터로 직접 그린다.** 외부 의존성이 0이고 어떤 변환 도구를 거쳐도 살아남는다.

## 기하

```
  ┌ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┐   지면 = 재단 + 2×margin,  margin = 도련 + 표시 길이
  │  ╔═════════════╗  │   도련: 배경이 넘어오는 구간
  │  ║ ┌─────────┐ ║  │   재단선: 논리 캔버스 0,0,W,H
  ─┼─ ║ │         │ ║ ─┼─  재단 표시는 도련 **바깥**에서 시작
```

`viewBox` 원점을 `-margin`으로 밀어 **기존 노드 좌표를 하나도 건드리지 않았다.** 재단선은 여전히 (0,0)이다.

재단 표시가 도련 안에 들어가면 그 선이 작품과 함께 인쇄된다. 그래서 표시는 도련 끝에서 시작한다. 선 길이는 도련 폭과 같게 뒀다 — **이 값의 출처는 없다.** 도련이 규격이 정한 유일한 "재단선 바깥 여유" 치수라 그것을 기준으로 삼았고, 그 사실을 코드 주석에 남겼다.

## 무엇을 바꿨나

- **`printSheetGeometry(blueprint)`** — 도련·표시 길이·지면 크기를 규격 mm에서 유도한다. 화면용이면 `null`.
- **`exportSvg`가 인쇄용일 때만 지면을 넓힌다** — 화면용 산출물은 바이트 단위로 예전과 같다(테스트로 고정). 인쇄 요건을 화면 산출물에 얹지 않는다.
- **캔버스를 덮는 배경 도형이 도련까지 확장된다** — exporter 자체 backdrop뿐 아니라 장면의 배경 노드도 함께. 배경 색이 캔버스 토큰과 다를 때 도련 구간만 다른 색이 되는 걸 막는다.
- **재단 표시 8선** — 네 모서리 × (가로·세로). 검정 stroke, 굵기는 도련의 1/12(최소 1px).
- **`blueprintOf` export** — exporter가 프로젝트에서 청사진을 되찾아야 규격을 알 수 있다.

## 게이트가 exporter를 되읽지 않는다

`check-export-artifacts`의 새 `checkSheetGeometry`는 **`printSpecs`의 mm과 청사진 치수만** 가지고 기대값을 따로 계산한다. `printSheetGeometry`를 호출해 비교하면 같은 코드가 같은 말을 하는 검사가 된다.

검사 항목: 지면 크기, `viewBox` 원점, 배경의 도련 확장, 표시 8개, 그리고 **화면용 산출물에 표시가 새어 들어갔는지**.

## finding 큐

- 재단 표시 선 길이의 출처가 없다 — 도련 폭을 관례로 썼다. 발주 전 인쇄소 확인 대상.
- 인쇄용 HTML은 그대로다. 도련·표시는 SVG 경로에만 있다(HTML은 화면 미리보기 용도).
- 배경 rect가 두 번 그려진다(exporter backdrop + 장면의 배경 노드). 이번 변경 이전부터 있던 중복이다.
- 논리 px → 300dpi 래스터화는 여전히 범위 밖. 지금 산출물은 벡터라 크기 무손실이지만, 발주처가 PDF를 요구하면 변환 단계가 필요하다.
