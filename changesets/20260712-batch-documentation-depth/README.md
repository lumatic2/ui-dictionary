## Amendments

1. Two YAML flow-mapping pitfalls hit during Adapt, both in unquoted `sources: [{ source_id: ..., note: ... }]` entries: a `note` value containing a comma inside parentheses (the `callout` enrichment) and a `note` value with a bare comma splitting a sentence (the `versioned-docs-switcher` enrichment) both parsed successfully through `validate-ui-vocabulary.py`'s Python YAML loader but then failed at `tsc` with "Object literal may only specify known properties" on the generated `terms.generated.ts` — the comma was read as introducing a second flow-mapping key rather than sentence punctuation. Both were fixed by rephrasing the note to avoid the embedded comma, consistent with the standing "no embedded punctuation inside unquoted flow scalars" guidance from prior batches, but this is the first time the failure surfaced downstream at the TypeScript build stage rather than at the Python validator — worth flagging that a comma inside an unquoted flow-mapping value can silently pass Python-side YAML validation and only break later.
2. Confirmed a `related:` structural rule not previously documented: the target `id` must be a `terms.yml` id, not a recipe id. The `api-error-reference-table` candidate's own `dedup_hints` pointed at `api-reference-three-column-layout` (a valid dedup signal, but a recipe id) — `validate-ui-vocabulary.py` rejected it as an "unknown related id" when used as a `related` target. Retargeted to `badge`, which is both a real term and the component actually reused by the new term's status column.
3. This batch is the first to reproduce internal-tools-depth's own Amendment 1 recommendation cleanly: staging the inbox and running the dedup audit strictly before authoring any recipe/term files produced a clean pre-adapt audit (exit 0, no self-match errors), unlike the `commerce-depth` and `internal-tools-depth` batches where recipes/terms were authored first and the audit showed expected self-matches.
