# Component Restyle Guide — 부품 층을 프로젝트 토큰으로

Date: 2026-07-17
Milestone: VI5 (plan: `plans/2026-07-17-vi5-component-layer-absorption.md`)
지위: 에이전트용 절차 가이드. shadcn/ui류 컴포넌트 배포 계층("부품 층")을 쓰되 "shadcn 룩"으로 수렴하지 않게 하는 재스타일 계약.

Audience: 코드를 생성하는 에이전트(Codex, Claude Code).

## 전제 — 스택에서 부품 층의 자리

Tailwind(문법) → **shadcn류(부품)** → 판단(recipe·signature). 부품 층은 동작 계약(Radix 접근성, 상태 머신)이 검증된 부품을 주지만 미학적으로 중립이라, 기본값 그대로 쓰면 모든 제품이 같은 얼굴("shadcn 룩")이 된다. 이 가이드는 부품의 *동작*은 유지하고 *look*만 프로젝트 소유로 되돌리는 절차다.

**대원칙 (DF-3 교훈)**: 재스타일의 목적지는 언제나 **작업 프로젝트의 토큰**이다. 이 가이드는 어떤 팔레트도 제안하지 않는다 — askewly 토큰 포함 어떤 외부 팔레트도 주입 금지. look의 정본은 프로젝트 DESIGN.md/토큰 SSOT.

## 재스타일 절차 (컴포넌트당)

1. **동작 계약 확인**: 그 부품이 보장하는 것(포커스 트랩, aria, 키보드 경로, 상태)을 recipe/문서로 파악 — 이건 절대 건드리지 않는다.
2. **look 표면 식별**: 색·radius·shadow·spacing·typography가 지정되는 지점을 나열 (shadcn이면 CSS 변수 정의부 + variant 클래스).
3. **토큰 재매핑**: 각 지점을 프로젝트 semantic 토큰으로 치환. shadcn CSS 변수 체계(`--primary`, `--radius` 등)를 쓴다면 **변수 정의만** 프로젝트 값으로 바꾸고 컴포넌트 파일은 그대로 두는 것이 최소 수술.
4. **정체성 지점 선택**: 전부 바꾸려 하지 말 것. 제품의 얼굴이 되는 2~3곳(주 버튼 형태, radius 성격, 포커스 링, 액센트 운용)만 뚜렷하게 — 나머지는 중립 유지가 오히려 세련.
5. **시그니처 판정**: 결과를 style-signature(원칙 5·비선호 5)로 자가 판정 — 특히 "대충 그린 CSS"(기본값 그대로 방치)와 "액센트는 신호"(변경 지점 과다) 사이 균형.

## "shadcn 룩 탈출" 체크리스트

기본값 흔적이 남는 대표 지점 — 하나도 안 바꿨다면 그 제품은 아직 부품 층의 얼굴을 쓰고 있다:

- [ ] radius 성격: 전 컴포넌트 동일 `--radius` 기본값(0.5rem) 그대로인가 — 제품의 기하 성격(각짐/둥긂)을 결정했는가
- [ ] 주 버튼: 기본 `bg-primary` 사각형 그대로인가 — 형태·무게가 브랜드를 담는가
- [ ] 포커스 링: 기본 ring 색·두께 그대로인가 — 접근성 유지하되 제품 색으로
- [ ] 그림자·보더 층위: 기본 `shadow-sm` 일괄인가 — 표면 위계가 제품의 밀도 철학을 따르는가
- [ ] 타이포: 기본 폰트 스택인가 — 최소한 디스플레이/본문 축은 프로젝트가 선택했는가
- [ ] 다크 모드: 기본 반전 값인가 — 다크에서의 브랜드 표현을 검토했는가

## 하지 말 것

- 동작 계약 수정(포커스·aria·키보드) — look 문제를 동작 코드로 풀지 않는다.
- 팔레트 발명 — 프로젝트에 토큰이 없으면 재스타일 전에 토큰부터(entry-protocol 1단계).
- 전면 개조 — 모든 컴포넌트를 다 바꾸면 부품 층을 쓰는 이유가 사라진다. 정체성 지점만.
- 인라인 리터럴로 덮어쓰기 — 재매핑은 토큰→토큰이지 토큰→hex가 아니다.
