# 최종 보고서 — DOG3 타이포 단계를 센다

> 2026-07-22 · horizon `design-output-gates` · milestone DOG3 · changesets 254·255·256 (+257 릴리스 후속)

## 1. 문제 정의

색은 잴 수 있게 됐는데(DOG1) 타이포는 못 쟀다. "한 화면에 글자 크기가 몇 종류인가"는 결정론적으로 잴 수 있는 항목인데 모델 자가 판정에 맡겨져 있었다.

Kraft(당근)의 `typography` 채점기가 같은 자리를 본다 — "한 화면 내 타이포 단계는 4단계 이내"(t1~t10 척도).

## 2. Objective 연결

Objective의 성공 상태 중 "의도적으로 디자인된 느낌"의 상당 부분이 타이포 위계다. 크기가 15종류인 화면은 위계가 없는 화면이다. 그리고 **이건 사람 눈보다 기계가 정확히 센다.**

## 3. 경로

3 step: 정규화 규칙 → CLI 배선 → 자기 자산 실측.

step-1이 이 milestone의 실질이다. 세는 것은 쉽고 **무엇을 한 단계로 볼지 정하는 것이 어렵다.**

## 4. 구현 결과

**정규화** — font-size는 네 형태로 나타난다(이름 유틸리티·임의값·인라인 style·CSS 선언). 전부 **px 정수로 환원하고 distinct set**을 센다.

**SSOT 우선** — 스케일 안 이름은 토큰 SSOT를 읽는다. 우리 값과 Tailwind 기본값이 겹치는 이름에서 다르기 때문이다(`lg` 20 vs 18, `xl` 28 vs 20, `2xl` 40 vs 24). 우리 값이 이긴다. 스케일 밖 이름만 Tailwind 조회 테이블로 간다.

**반응형 합침** — 한 시점에 한 브레이크포인트만 렌더되므로 `md:text-lg`는 별도 단계가 아니다.

**임계값 5** — Kraft는 4다. 우리는 5다. **우리 토큰 스케일이 5단계라, 4를 쓰면 디자인 시스템의 스케일을 온전히 쓴 화면이 위반이 된다.** 남의 숫자를 우리 토큰에 대조하지 않고 옮겼으면 규칙이 자기가 지켜야 할 시스템과 모순됐다.

**CLI** — `typography-scale-exceeded` 규칙, 보고 줄은 임계값을 깨뜨린 값이 처음 등장한 줄, `--typography-threshold` 옵션. 계수는 파일 단위이고 DOG1의 마스킹을 재사용해 주석·SVG 안 크기는 세지 않는다.

## 5. 이슈와 해결

### Failure probe가 다시 자기 무용함을 드러냈다

step-1의 probe B(반응형 접두사 미처리)가 **통과했다.** `responsive.tsx`에 `md:text-lg`와 순수 `text-lg`가 둘 다 있어, 접두사 처리를 통째로 없애도 뒤엣것이 같은 20px을 채웠다. **테스트가 그 코드를 지키지 않고 있었다.**

`variant-only.tsx`를 신설해 20·28에 접두사 없이는 도달할 수 없게 만들었고, 다시 돌리니 정확히 그 테스트 1건이 실패했다.

이 horizon에서 probe가 무용함을 드러낸 **두 번째** 사례다(DOG1 step-2는 치환 자체가 no-op이었다). 규율이 값을 하고 있다.

### 릴리스에서 CI 검증 결함 2건 (changeset 257)

`0.2.0` 릴리스에서 **배포는 성공했는데 검증 단계가 실패**했다.

- **패키지 존재만 확인하고 버전은 안 봤다** — `HTTP 200`은 0.1.0 때문에 즉시 통과, 정작 0.2.0은 전파 전(`ETARGET`). DOG2의 교훈을 **한 칸 얕게** 적용했다.
- **exit code 오독** — `npx` 실패와 린터 위반이 둘 다 exit 1이라, 한 원인이 `bad` 검사에서는 거짓 안심을, `clean` 검사에서는 거짓 경보를 냈다.

수정: 버전 단위 조회 · 설치와 검사 분리 · exit code 명시 비교.

## 6. 결과물과 증거

**크기 회고**: changeset **3개**(254·255·256)로 닫혔다. 선언(`changesets>=3`)과 일치 — 인플레·디플레 없음. 릴리스 후속 257은 milestone 밖 정비다.

**자기 자산 실측** (99개 파일):

```
단계수:  0→25  1→35  2→21  3→7  4→2  5→1  |  6→1  7→1  8→2  11→2  15→1  16→1
```

91.9%가 5 이하. 초과 8건 중 **5건이 400줄 초과 다중 컴포넌트 파일**(`App.tsx`는 23,508줄), **4건이 12px 미만 장식용 미니어처**. 400줄 이하만 보면 89개 중 3개(3.4%).

**판정: 임계값은 틀리지 않았다. 경계가 틀렸다.** 23,508줄 파일이 15단계를 쓰는 건 그게 화면 하나가 아니어서지 타이포 규율이 무너져서가 아니다. 임계값을 올려 통과시키면 진짜 화면의 남용까지 함께 통과한다.

- 실표면: 공개 배포본이 실표면이다. **레포 밖 임시 디렉터리**에서 npm 공개 레지스트리의 `@askewly/design@0.2.0`을 설치해 실행 — 6단계 fixture `six-steps.tsx:10 [typography-scale-exceeded] font-size steps: 12, 14, 16, 20, 28, 40 (6 > limit 5)` (exit 1), 5단계 경계 fixture PASS (exit 0), `--typography-threshold 8` PASS. 아울러 `0.2.0`은 **provenance attestation**(`slsa.dev/provenance/v1`)을 갖는다 — 토큰 없이 OIDC로 배포됐다는 증거.
- 재현: `cd packages/cli && npm run build && npx vitest run` → 48 passed; `npx tsc --noEmit` → 0; 배포본 검증은 임시 디렉터리에서 `npm install @askewly/design@0.2.0` 후 `node_modules/.bin/askewly-design verify <dir> --ext tsx`.

증거: `evidence/design-output-gates/dog3-typography.md`

## 7. 후속 제안

- **파일 = 화면 경계의 한계** — 다중 컴포넌트 파일에서 오탐이 난다. 해소하려면 "화면"을 컴포넌트·라우트 단위로 잡아야 하고 그건 별도 설계 문제다. **DOG7 사람 관측에서 "오탐이 성가신가"의 1순위 대상.**
- **장식용 미니어처(<12px) 제외 여부** — 초과 8건 중 4건이 해당하나, "장식"을 크기만으로 기계 판정할 근거가 없어 규칙을 만들지 않았다. 사람 관측 입력으로 남긴다.
- **수정된 CI 검증 경로는 아직 CI에서 안 돌았다** — 로컬 실증만 했다. 다음 릴리스가 그 게이트다.
