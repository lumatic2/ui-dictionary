# 최종 보고서 — ECT1 토큰이 조회되고, 실재가 검증된다

> 완료: 2026-07-21 · 대상: ECT1 · 작성: 완료 경계(§B3)
> 배치: `archive/reports/2026-07-21-ect1-token-lookup-and-validation.md` (record — 작성 후 동결)

## 1. 문제 정의

직전 horizon의 마지막 관측에서 사용자가 편집기로 색을 바꾸지 못했다. 원문 그대로:
*"요소 고르고 크기 바꾸는건 알겠는데 색은 어떻게 바꾸는지 모르겠네."*

관측 직후 계측하니 화면 전체에 "색"이라는 단어가 0건, 색 견본이 0개였다. 색을 바꾸는 유일한
경로가 자유 텍스트 입력 하나였고, 유효한 토큰 이름을 미리 알고 오타 없이 타이핑해야 했다.
목록도 견본도 자동완성도 없었다.

ECT1은 그중 **화면이 아니라 그 아래**를 맡았다. 색 선택기를 지으려면 "이 문서가 쓸 수 있는
토큰이 뭐냐"를 물어볼 곳이 있어야 하는데, 그 질문을 받을 API 자체가 없었다. 그리고 검증이
토큰의 실재를 보지 않아 오타가 조용히 저장되고 있었다.

## 2. Objective 연결

북극성의 "사람이 코드 네이티브 캔버스에서 실제 컴포넌트를 조작하고 에이전트와 무손실
왕복한다"는 축이다. 이번에 움직인 건 그 축의 **하부 계약** — 사람과 에이전트가 같은 문서를
고칠 때 토큰이 지켜지는가. 사람 쪽 UI는 ECT2가, 에이전트 쪽 표면은 이번에 닫혔다.

## 3. 경로

horizon `editor-color-and-token-editing`을 개설하고 그 첫 milestone으로 ECT1을 실행했다.
계획한 step 3개(열거 API → `kind` 메타데이터 → 실재 검증)를 순서대로 밟았다.

**계획과 실제가 갈린 지점이 둘 있다.** 첫째, step-1 검증이 계획의 전제를 반증했다 —
"두 어휘의 이름은 겹치지 않는다"고 적었는데 실제로는 겹쳤다. 둘째, step-3 완료 후
독립 검증이 두 번 연속 refuted를 냈고, 그 결과 계획에 없던 네 번째 작업(길목 재설계)이 붙었다.

## 4. 구현 결과

편집기가 이제 "이 문서가 쓸 수 있는 토큰이 뭐냐"고 물어볼 수 있다. 답에는 토큰 이름과
실제 색 값, 그리고 그게 색인지 글꼴인지가 함께 온다 — ECT2의 색 선택기가 그 위에 선다.

없는 토큰은 더 이상 저장되지 않는다. 사람이 인스펙터에서 오타를 쳐도, 에이전트가 라이브
브리지로 문서를 고쳐도 같은 규칙에 걸리고, 사유가 사람이 읽는 문장으로 화면에 뜬다.
이미 저장돼 있던 죽은 바인딩은 그대로 열린다 — 저장은 막되 사용자가 자기 문서를 잃지 않는다.

## 5. 이슈와 해결

**계획의 전제가 틀렸다.** 두 토큰 어휘가 이름을 공유한다(`text.muted`·`text.secondary`가
양쪽에 있고 값만 다르다). 코드 주석도 "겹치지 않는다"고 단언하고 있었는데 사실이 아니었다.
격리는 이름이 아니라 출처로 이뤄진다는 걸 확인하고, 주석과 horizon 닫는 기준 3의 관측 방법을
함께 고쳤다. 원래 기준("이름 교차 0건")으로는 잴 수 없는 것이었다.

**독립 검증이 두 번 refuted했다. 이게 이 milestone의 핵심 사건이다.**
1차: 검증 게이트가 `set-node-property`만 막고 있었고 `update-node.patch.tokenBindings`가
뚫려 있었다. 검증자는 재현 테스트로 실제 가짜 토큰을 문서에 박았다. 막은 쪽이 사람 경로,
뚫린 쪽이 에이전트 경로였다 — 이 milestone이 겨눈 위협 모델이 정확히 후자였으므로
부수적 누락이 아니라 과녁을 빗나간 것이었다.
2차: 그 경로를 막았더니 `create-node`가 뚫렸다. 노드 내용을 통째로 신뢰하고 있었다.

**두 번째에 방법이 틀렸다는 게 드러났다.** 뚫린 경로를 하나씩 막는 구조는 새 연산이 생길
때마다 샌다. 자기 게이트는 자기가 막은 경로만 두드리기 때문에 스스로는 이걸 못 잡는다.
그래서 모든 연산이 이미 지나는 길목으로 검사를 옮기고, 이 연산이 새로 넣거나 바꾼 바인딩만
보게 했다(전체를 보면 기존 죽은 바인딩까지 걸려 사용자가 자기 문서를 못 쓴다).

3차 검증이 confirmed를 냈다. 검증자는 요청 범위를 넘어 실제 브리지 HTTP 서버에 공격 요청을
쏘고, 연산 13종을 전수 조사하고, 소스 패치 경로까지 확인했다.

**해소하지 않고 남긴 판단이 하나 있다.** 스냅샷 로드는 일부러 검사하지 않는다. 막으면
토큰이 사라진 옛 문서를 사용자가 못 연다. 검증자가 이걸 우회로로 분류했으나, 3차에서
브리지에 문서를 업로드해 로드시키는 엔드포인트가 없음을 확인하고 "타당"으로 판정했다.
쓰기와 읽기의 구분을 코드 주석에 명시해 다음 사람이 누락으로 오해하지 않게 했다.

## 7. 후속 제안

- **ECT2(색이 색으로 보인다)로 즉시 연쇄** — 열거 API와 `kind`가 준비됐으므로 선택기가 그 위에 선다.
- **ECT4의 "허용 키 정본은 하나"를 조건으로 격상** — 이번에 같은 병리(규칙이 여러 곳에 흩어짐)를
  실물로 두 번 맞았다. ECT4에서는 단일 출처를 선택지가 아니라 통과 조건으로 잡는다.
- **어휘 등록의 단일 방어선** — `registerTokenVocabulary` 미등록 시 모양 검사로 조용히 강등된다.
  배선 테스트 하나가 유일한 방어선이라, 등록을 구조적으로 강제하는 방안은 결정 사안으로 남긴다.
- **생성기의 중복 표현** — `SEMANTIC_COLOR_MAPPINGS`(손으로 적은 19줄)와 `buildSemanticColorMap`
  (순회)이 같은 집합을 두 방식으로 표현한다. ECT4와 성격이 같아 함께 볼 것.
- **자체 검증의 구조적 한계** — 이번에 자체 probe 11건이 전부 초록인 상태에서 독립 검증이 두 번
  뚫었다. 다음 milestone도 완료 주장 전 독립 검증을 거친다(이번 경험이 그 비용을 정당화했다).
