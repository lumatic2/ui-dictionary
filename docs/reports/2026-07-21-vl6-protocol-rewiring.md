# 최종 보고서 — VL6 프로토콜 재배선

> 완료: 2026-07-21 · 대상: VL6 (horizon `vocabulary-in-use`) · 작성: 완료 경계(§B3)

## 1. 문제 정의

VL2~VL5가 자산을 다 만들어놨다 — 사전 562개가 배포됐고, 자산 매핑이 있고, 폴백 경로가 있고, 판별 군집 6개가 검증기를 통과한다. 그런데 **프로토콜이 그중 아무것도 부르지 않았다.** VL1 계측: `entry-protocol.md`의 용어 언급 0회, 요소가 미정일 때의 결정 단계 없음, 레시피가 없을 때의 경로 없음.

프리모템 시나리오 2가 정확히 이것이었다 — "자산 15개를 잘 만들고 링크 한 줄을 걸었는데 에이전트는 여전히 분류→레시피로 직행한다."

## 2. Objective 연결

만든 것과 쓰이는 것 사이의 마지막 배선이다. 이 milestone 전까지 이 horizon의 산출물은 전부 "있지만 안 불리는 것"이었다.

## 4. 구현 결과

**A/B/C 분기 앞에 세 단계가 생겼다**: N-1 용어 조회 → N-2 요소 결정 → N-3 자산 획득(없으면 폴백). 순서에 이유가 있다 — 요구가 서술적일 때 먼저 어휘로 번역해야 후보군이 나오고, 후보가 있어야 축이 갈리며, 요소가 정해져야 자산을 찾는다.

**우회 경로를 닫았다.** 세 분기 위에 "요소가 미정인 채 여기 도착했으면 돌아가라 — 아래 어느 분기도 대신 정해주지 않는다"를 못 박았고, C 분기의 열린 끝("레시피가 있으면")을 폴백으로 연결했다. B 분기에는 "진단 결과가 *잘못된 요소를 썼다*면 그건 스타일 수정이 아니라 N-2 결정이다"를 넣었다.

**두 가지를 정상 경로로 규정했다.** 사전에 없는 개념이면 막히지 말고 흐를 것, 축을 답할 수 없으면 사용자에게 그 질문을 그대로 되물을 것. 둘 다 실패가 아니다.

**`요소 결정` 보고 블록**이 계약이 됐다. 고른 것·군집·갈린 축·기각한 대안·쓴 자산. 요소가 지정된 작업은 "해당 없음 — 요소 지정됨"이 정당한 값이다.

**배포 인덱스에 실렸다.** `llms.txt`에 `## Decisions` 섹션이 생겼고, 군집 파일은 디렉터리를 훑어 자동 등재된다.

## 5. 이슈와 해결

**자기 적용이 통했다.** VL3에서 손으로 판단해 만든 combobox를 새 프로토콜로 되짚었더니 세 단계가 같은 답에 도달했다 — 인덱스에서 2 fetch로 용어를 찾고, `option-set-known=open-ended` 축에서 combobox로 갈리고, 자산 매핑이 비어 있어 폴백으로 간다. 보고 블록의 모든 필드가 채워졌고 빈 칸이 0이었다.

**프로토콜이 길어지는 것이 리스크였다.** 신설 절이 기존 필수 단계(자가 판정·사람 확인)를 밀어내면 더 나쁜 결과가 된다. 위치를 대조해 `Always` 절이 여전히 앞에 있음을 확인했다.

**등재 생성기에 게이트를 하나 넣었다.** `decisions/README.md`가 없으면 등재를 거부한다 — 군집 파일만 실리고 인덱스가 없으면 에이전트가 진입점을 못 찾는다.
