# Design Brief v2 — 착수 인터뷰 계약 (결정 공간 지도)

Date: 2026-07-19 (v2 — 리서치 기반 전면 개정: `research/2026-07-19-studio-depth-brief-practice.md` · `-tool-practice.md`)
지위: 정본. 진입 프로토콜(entry-protocol.md)의 브리프 단계가 이 문서를 가리킨다.

시니어 디자이너는 의뢰를 받으면 바로 그리지 않는다 — 브리프를 받는다. 이 계약의 목적: ① 추정을 질문으로 대체한다 ② 답변을 프로젝트 `DESIGN.md`로 저장해 **룩 소유권을 만든다** ③ 시각 결정은 텍스트가 아니라 **실물로 보고 고르게** 한다(Brief Studio).

## 1. 규모 게이트 — 언제 인터뷰하는가

| 상황 | 판정 |
|---|---|
| `DESIGN.md`(또는 토큰 SSOT)가 이미 있음 | **인터뷰 없음** — 파일에서 파생, 파일이 답 못 하는 새 결정만 개별 질문 |
| 새 페이지·사이트·화면·제품 | **풀 브리프** — §2 전략층(텍스트) + §3 시각층(스튜디오) |
| 컴포넌트 1개·수정·폴리싱 | **생략** — 기존 스타일 파생, 무토큰이면 최소 질문 1개만 |
| "인터뷰 없이"·"알아서" 명시 | 생략 — 추천 기본값 + 보고에 추정 목록 명시 |

오발동 금지: 소형 작업에 질문·스튜디오를 들이미는 것은 게이트 실패다.

## 2. 전략층 — 텍스트 인터뷰 (업계 브리프의 코어)

선택지+추천 형식 유지. Claude=AskUserQuestion 묶음, Codex=평문 일괄. 건너뛴 답 = 추천값 채택으로 기록. (※표시 항목은 본질상 열린 질문 — 짧은 주관식 허용.)

1. **정체성·톤** — 누구를 위한 무엇, 톤 후보(따뜻함/신뢰·정제/미니멀/힙·에너지).
2. **브랜드 인지 gap** ※ — "지금 어떻게 인식되는가 vs 어떻게 인식되고 싶은가" (업계 설문 공통 핵심 축).
3. **경쟁·포지셔닝** — 경쟁/참고 사이트 2~3개와, 그들 대비 어느 쪽으로 설 것인가(전통적↔진보적 / 기업적↔인간적 축 제시).
4. **비주얼 키워드** ※ — 형용사 후보 목록에서 3~4개 수렴 (Toptal 20→4 압축 관행의 축약판). 이 키워드가 §3 스타일 타일 후보 생성의 입력이다.
5. **룩 소유권** — 기존 브랜드 자산(로고·색·서체) 유무. 있으면 파생, 없으면 §3으로.
6. **성공·범위** — 이 화면이 이루려는 것(KPI 한 줄), 화면 목록, 실데이터 vs 더미, 다크모드(기본 제공), 접근성 레벨(기본 WCAG AA).

## 3. 시각층 — Brief Studio 2단 구조 (기본 경로)

정본 절차: [brief-studio.md](https://ui.askewly.com/llms/docs/design-system/brief-studio.md). 텍스트 질문은 스튜디오 구동 불가 시 폴백.

**1단 — 스타일 타일 (방향 선택)**: 색+타이포+형태+질감이 조합된 방향 타일 3~4개를 먼저 제시한다(Style Tiles 관행, 2011~). 비주얼 키워드(§2-4)에서 파생. 타일 선택은 2단 후보의 **추천 재정렬**로 작동한다 — 강제 필터 아님(타일 밖 조합 허용).

**2단 — 축별 세부 (그룹당 후보 6~8, 추천순 정렬)**:

| # | 축 | 후보 제시 형태 (리서치 관측 근거) |
|---|---|---|
| 1 | 배경/표면 베이스 | 미니 UI 프리뷰 카드 (웜 페이퍼/쿨 그레이/다크 등 온도·명도 변주) |
| 2 | 액센트 팔레트 | 같은 미니 UI에 액센트만 교체 — 베이스와 독립 선택 |
| 3 | 타이포 페어 | 헤딩/본문 **페어 단위** 스펙시멘 (Squarespace 14종 관행) |
| 4 | 페이지 구성 | 섹션 조합 미니 와이어프레임 — 12유형 카탈로그 + 업종 조건 노출(예약형 등), brief-studio.md §3.5 |
| 5 | 헤더 | 실물 미니 렌더 (로고만/링크형/링크+CTA/센터드 등) |
| 6 | 푸터 | 실물 미니 렌더 (한 줄/멀티 컬럼/연락 중심/뉴스레터) |
| 7 | 카드 스타일 | 실물 렌더 (플랫+명도/보더/소프트 섀도/사진 상단형) |
| 8 | 그리드·밀도 | 3~5 스텝 프리셋 슬라이더 (도구 관행: Lovable·Figma) |
| 9 | radius·형태 언어 | 3~5 스텝 프리셋 슬라이더 (사각→라운드→필) |
| 10 | 이미지 스타일 | 실물 썸네일 — 실사(스톡)/일러스트/3D + 다큐멘터리 vs 애스퍼레이셔널 톤 (ybug Q17·Smashing) |
| 11 | 아이콘 스타일 | 실물 렌더 (outline/filled/duotone/없음) — 이모지 아이콘은 후보 금지(비선호) |
| 12 | 다크모드 | 토글 + 다크 미리보기 (Stitch 관행: 별도 토큰 세트 미러링) |
| 13 | 모션 톤 | 만져보는 데모 (담백/표준/풍부) — **업계 브리프 4종 전부 미질문(공백) — 우리 차별화 축**, expressive-stack 티어 연결 |
| 14 | 접근성 레벨 | AA(기본 추천)/AAA — 업계 설문 4중 1만 질문(공백) — 명시 확인 |
| 15 | 히어로 헤드라인 (ST4) | 카피 후보 3~6안 실물 타이포 렌더 — 전략층 키워드 파생, 미리보기 즉시 반영 |
| 16 | 서브 카피 (ST4) | 톤 변주 후보 (설명/가치/압축/증거 등) |
| 17 | CTA 구성 (ST4) | 개수·위계 실물 렌더 (주+보조 / 하나 집중 / 링크형 등) |
| 18 | 마우스 인터랙션 (ST4) | 만져보는 데모 — 부양/틴트/글로우/밑줄 확장/자석(수동 승인) |

축마다 후보는 §2 답변(톤·키워드)의 파생이어야 한다 — 무작위 큐레이션 금지. 실험적 표현(모션 "풍부"의 ③④ 티어)은 수동 승인 원칙 유지.

## 4. DESIGN.md 저장 계약

- 풀 브리프 결과는 프로젝트 루트 `DESIGN.md` — **Google Stitch 공식 스펙**(google-labs-code/design.md, alpha: frontmatter `name/colors/typography/rounded/spacing/components` + 고정 8섹션 + `{path.to.token}` 참조, `colors.primary` 필수). 출발점 `templates/DESIGN.md.tmpl`.
- 전략층 답변은 Overview에, 축 선택은 대응 토큰·섹션에(구성·헤더·푸터·그리드=Layout, 카드·아이콘=Components, 이미지 스타일·모션 톤·접근성=Do's and Don'ts 또는 해당 섹션 프로즈).
- **결정 vs 추정 구분 기록** 의무 — 사용자가 명시한 것과 추천값 채택을 다음 세션이 구분할 수 있어야 한다.

## 5. 대화 불가 환경(headless) 폴백

인터뷰·스튜디오 시도로 멈추지 말 것: 규모 게이트만 적용, 풀 브리프 상황이면 전 축을 추천 기본값으로 진행하고 보고에 "브리프 미실시 — 추정 목록"을 명시한다.
