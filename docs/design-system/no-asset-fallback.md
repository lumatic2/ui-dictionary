# No-Asset Fallback — 자산 없는 용어를 구현하는 경로

Date: 2026-07-21
Milestone: VL3 (plan: `plans/2026-07-21-vl3-reference-repair.md`)
지위: **레시피도 코드 자산도 없는 용어를 골랐을 때의 정본 경로.** `entry-protocol.md`의 자산 획득 단계가 자산 없음 분기에서 이 문서를 가리킨다.

## 왜 이 문서가 있나

용어 562개 중 **레시피가 있는 것은 81개(14.4%)**, 코드 자산까지 있는 것은 52개다. 나머지 **481개는 아무 자산도 없다**(계수: `docs/ui-vocabulary/term-asset-map.json`).

판정이 정확해져도 이 481개 중 하나를 고르면 만들 근거가 없다. 그러면 에이전트는 "combobox를 쓰세요"까지만 하고 멈추거나, 아무 근거 없이 감으로 그린다. 사용자 입장에서는 둘 다 아무것도 안 바뀐 것과 같다.

**자산이 없다는 것은 재료가 없다는 뜻이 아니다.** 용어 항목 자체가 재료다.

## 먼저 확인할 것

`docs/ui-vocabulary/term-asset-map.json`에서 그 용어를 찾아라. `recipes`와 `code_assets`가 둘 다 비어 있을 때만 이 문서의 경로를 쓴다. 하나라도 있으면 그쪽이 우선이다(코드 자산 > 레시피 > 폴백).

## 재료 목록

용어 항목에서 이 네 가지를 뽑아 구현의 출발점으로 삼는다. **감으로 시작하지 않는다.**

| 재료 | 필드 | 무엇을 정하나 |
|---|---|---|
| 부위 | `visual_anatomy` | 무엇이 화면에 있어야 하는가 — 빠뜨리면 그 용어가 아니다 |
| 경계 | `when_to_use` · `anti_use` | 이 구현이 넘지 말아야 할 선 |
| 이웃 | `related` | 헷갈리는 옆 용어와 무엇이 달라야 하는가 |
| 구조 참고 | 같은 그룹의 인접 레시피 | 자산이 *있는* 이웃의 뼈대 — 베끼는 게 아니라 구조를 참고한다 |

## 절차

1. **부위를 체크리스트로 만든다.** `visual_anatomy` 항목을 그대로 옮겨 적고, 구현이 끝난 뒤 하나씩 대조한다. 이게 "그 용어를 만들었다"의 최소 기준이다.
2. **경계를 반대 체크리스트로 만든다.** `anti_use` 각 항목에 대해 "내 구현이 이걸 범하는가"를 묻는다. 범하면 용어를 잘못 골랐거나 구현이 샌 것이다.
3. **이웃과의 차이를 한 줄로 적는다.** `related`의 `compare` 항목을 보고, 내 구현이 그 이웃이 아니라 이 용어인 이유를 쓴다. 못 쓰면 두 용어가 구현상 같아졌다는 뜻이다.
4. **구조는 인접 레시피에서 참고한다.** 같은 그룹에 레시피가 있으면 그 anatomy·states·checks 절을 읽고 뼈대를 빌린다. **없으면 없는 대로 간다** — 존재하지 않는 레시피를 만들라고 요구하지 않는다(그러면 폴백이 아니라 순환이다).
5. **상태를 빠뜨리지 않는다.** 용어 항목에 상태 목록이 없어도 상호작용 요소라면 hover·focus·disabled·error·empty를 스스로 챙긴다 — 이건 `principles.md`의 공통 계약이지 용어별 사항이 아니다.
6. **토큰은 작업 프로젝트 것을 쓴다.** 폴백이라고 해서 색·간격을 지어내도 되는 게 아니다.
7. **보고에 남긴다.** 어느 용어를 폴백 경로로 만들었는지, 부위 체크리스트를 몇 개 충족했는지. 이 기록이 "어느 용어에 자산이 급한가"의 입력이 된다.

## 하지 않는 것

- **레시피가 없다고 멈추지 마라.** `entry-protocol.md` C 분기의 "Fetch the matching recipe if one exists"가 열린 채 끝나던 것을 메우는 게 이 문서다.
- **자산이 없다고 감으로 그리지 마라.** 재료 4종이 있는데 안 쓰는 것은 폴백이 아니라 방치다.
- **폴백 산출물을 레시피라고 부르지 마라.** 레시피는 `recipe-format.md` 계약을 갖춘 정본 자산이다. 폴백 구현이 반복해서 쓸 만하다고 판단되면 그때 정식 레시피로 승격을 제안한다.

## 실증 — `combobox` (자산 없는 481개 중 하나)

이 문서의 경로를 실제로 밟아 만든 결과다. 산출: `examples/ui-vocabulary-site/src/components/fallback/combobox.tsx`

**부위 대조** (`visual_anatomy` 4/4):

| 부위 | 구현 |
|---|---|
| input trigger | 검색어를 입력하는 인풋 자체가 트리거 — 포커스 시 목록이 열린다 |
| typed query | 입력값이 그대로 보이고, 그 값으로 목록이 좁혀진다 |
| filtered option list | 질의에 맞는 항목만 렌더 (`listbox` 역할) |
| highlighted option | 키보드 이동 시 활성 항목이 시각·`aria-activedescendant` 양쪽으로 표시된다 |

**경계 대조** (`anti_use` 2/2 미위반):

- "선택지가 적으면 select나 radio group이 더 단순하다" → 데모 옵션이 10개 이상이고, 컴포넌트 주석에 소수 옵션이면 다른 요소를 쓰라고 명시했다.
- "자유 입력만 필요하면 text field를 쓴다" → 목록 밖 임의 문자열을 값으로 확정할 수 없다. 반드시 목록의 항목이 선택되어야 값이 정해진다.

**이웃과의 차이** (`related`):

- `select`가 아닌 이유: 입력으로 목록을 좁힌다. select는 좁히지 않는다.
- `dropdown-menu`가 아닌 이유: 값을 정하는 control이지 명령 목록이 아니다. 항목 선택이 즉시 동작을 실행하지 않고 **값**이 된다.

**구조 참고**: 같은 `input-pickers` 그룹에 레시피가 없어 4단계는 건너뛰었다 — 폴백은 없는 대로 간다.
