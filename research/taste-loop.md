# Taste Loop — 판단 상향 흡수 계약

Date: 2026-07-17
Milestone: TC1 (plan: `plans/2026-07-17-tc1-taste-loop-contract.md`)
지위: 최상급 레퍼런스에서 **판단**을 흡수하는 절차 계약. `research/reference-loop.md`(RL — 커버리지 흡수: 신규 recipe/term)를 대체하지 않고 증설한다 — RL의 산출물은 *새 항목*, taste loop의 산출물은 *기존 판단 자산의 갱신*이다. 관찰 중 신규 recipe 후보가 나오면 RL inbox로 넘긴다(이 계약의 소관 아님).

## 핵심 계수 규칙 (이 계약의 정체성)

**갱신 없는 관찰 = 미소화.** 관찰 1건은 아래 판단 자산 4종 중 하나 이상의 *구체적 갱신*(신규 클러스터·행·원칙 개정·recipe 절 보강)으로 끝나야 흡수로 계수한다. "좋더라"로 끝난 관찰은 ledger에 `미소화`로 표기하고 Close Criteria 계수에서 제외한다 — 단 기록 자체는 남긴다(재소화 후보).

갱신 대상 자산 4종:

1. `docs/design-system/style-signature.md` — 운용 원칙·비선호 (개정은 사용자 확인 필수 — 시그니처는 사용자 인터뷰 확정본)
2. `docs/design-system/anti-patterns.md` — 클러스터 추가·보강
3. `knowledge/expressive-stack.md` — 결정 표 행·판정 규칙
4. `recipes/**/*.md` — 기존 recipe의 Checks/Anti-patterns/Agent notes 절 보강

## 관찰 스키마 (7필드 — 초과 금지, RL 실증 교훈: 장부 피로가 루프를 죽인다)

| 필드 | 의미 |
|---|---|
| `id` | `T-<번호>` |
| `source` | 제품/표면 + URL + 접근일 (전역 인용 규칙) |
| `관찰` | 무엇을 봤나 — 한 문단, 스크린샷 경로 포함 |
| `원리` | 그 판단이 *왜* 좋은가 — 소스와 독립적으로 서술(이식 가능해야 함) |
| `diff` | 우리 판단 자산과의 차이 — "이 판단이 우리 어디에 없나/충돌하나" |
| `갱신` | 실제 반영한 자산·위치·changeset (없으면 `미소화 — <사유>`) |
| `게이트` | 성립성 테스트 결과 (제품 실화면은 `해당 없음`, 콘셉트 소스는 필수) |

## 성립성 게이트 (콘셉트 소스 — Dribbble·목업·수상작 전용)

콘셉트 아트는 제약 없는 그림이라 판단 오염 위험이 있다. 4문항을 통과한 원리만 흡수한다:

1. **실데이터**: 긴 문자열·빈 목록·수백 행에서도 이 룩이 성립하는가?
2. **상태**: 에러·로딩·비활성 상태가 이 디자인 언어로 표현 가능한가?
3. **한글**: 한글 타이포(줄바꿈·자간 특성)에서도 성립하는가?
4. **다크모드·접근성**: 대비·포커스가 유지되는가?

FAIL 시: 흡수하지 않되, "왜 안 성립하는가"를 anti-patterns 재료로 역이용할 수 있다(그 경우 갱신 필드에 anti-pattern 반영을 기록 — 이것도 흡수로 계수).

## 절차 (관찰 1건 기준)

1. **관찰**: 실서비스를 브라우저로 직접 본다(스크린샷 증거). 로그인 벽 뒤 표면은 사용자 브라우저 경유 + 사용자 승인으로만.
2. **원리 추출**: 소스 이름을 지우고도 성립하는 문장으로 쓴다. "Linear의 사이드바"가 아니라 "보더 없이 배경 명도 2단계로 위계를 만든다".
3. **diff**: 판단 자산 4종을 grep — 이 원리가 이미 있으면(중복) 미소화 처리, 충돌하면 충돌 내용 기록.
4. **갱신**: 해당 자산에 반영 + changeset. 시그니처 개정만은 사용자 확인 후.
5. **장부화**: `research/taste/ledger.md`에 1행 append.

## 불변식

- **look 복사 금지** (DF-3): 흡수하는 것은 판단·원리이지 팔레트·스타일이 아니다. 갱신 문구에 특정 제품의 색·폰트 값이 들어가면 반려.
- 시그니처 개정은 에이전트 단독 불가 — 제안 → 사용자 확인 → 반영.
- 배포는 갱신된 자산의 기존 경로(llms 재생성)를 그대로 탄다 — taste loop 전용 배포면 없음.
