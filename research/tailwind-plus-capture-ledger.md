# Tailwind Plus capture ledger

Date started: 2026-07-01
Purpose: track capture-first parity work from public Tailwind Plus pages to UI Dictionary pages.

## Rules

- One public Tailwind leaf page gets one row.
- The implementation target is close structural parity first: navigation, page
  hierarchy, example rhythm, toolbar behavior, desktop/mobile layout, and public
  locked states.
- Runtime code, copy, template names, and assets must remain UI
  Dictionary-native. Tailwind screenshots are evidence, not app assets.
- If a Tailwind page is locked, paid-only, or requires sign-in for the main
  content, record the public shell and continue to the next public leaf.
- Do not block the whole phase on missing representative terms. Add a small
  placeholder term or mark `needs-term` and continue if the page shell can be
  implemented.
- Every implemented row needs desktop and mobile Tailwind capture plus desktop
  and mobile local capture before `verified`.

## Status Values

- `planned`: known target, not captured yet
- `captured`: Tailwind evidence exists
- `implemented`: local route exists
- `verified`: local desktop/mobile smoke passed
- `needs-term`: route can exist, but stronger representative terms are needed
- `locked-preview`: public preview/shell visible, code/detail paid or locked
- `paid-only`: no useful public page body beyond payment/sign-in
- `deferred`: intentionally skipped with reason
