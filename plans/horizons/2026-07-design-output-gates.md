# HORIZON — 산출물이 좋은지 기계가 잰다, 매체마다 다른 자로

> 생성: 2026-07-22 · Status: **approved 2026-07-22** — 사용자가 6 milestone 묶음과 미결 결정 4건(타이포 임계값 5·검사 범위 만진 파일만·license MIT·scope public)을 추천대로 승인. horizon 전체 연쇄로 진행.
> 진행 상태의 정본은 `ROADMAP.md` marker다. 후보 4(`design-output-static-linter`)와 후보 8(`medium-crossing-assets`)을 사용자 확정으로 **병합**해 개설한다.
> changeset 번호 구간 예약: **247~266** (병렬 horizon 번호 충돌 이력에 따른 사전 분할, 2026-07-21 교훈)

## 목표

Askewly Design의 품질 게이트에 두 개의 구멍이 있다.

1. **기계가 잴 수 있는 것을 모델 자가 판정에 맡긴다.** 사람 눈은 스크린샷에서 `#3B82F6`과 `var(--color-accent)`를 구분하지 못한다 — **렌더 결과가 똑같기 때문이다.**
2. **매체가 늘었는데 자가 없다.** Objective가 2026-07-21 확장되며 슬라이드·지면이 범위에 들어왔지만, 게이트는 전부 화면 UI를 전제한다.

이 horizon은 두 구멍을 **하나의 마무리 절차**에 함께 꽂는다. 병합한 이유가 여기다 — 둘 다 `askewly-design` 마무리 절차라는 같은 자리에 꽂히고, 따로 열면 두 horizon이 같은 파일을 동시에 고친다.

- **예상 분량: ~20 changeset** (7 milestone, step 트리 합산). 하한(≥10) 충족.

## 왜 지금 — 그리고 실사가 브리프를 뒤집었다

착수 리서치의 브리프(`research/2026-07-20-design-output-static-linter-brief.md`)는 §2에서 이렇게 단정했다:

> "세션 중 에이전트가 남의 프로젝트에 실제로 뱉은 **UI 코드**를 보는 검사기는 하나도 없다."

**틀렸다.** 직전 horizon의 교훈("실사 기본값은 실행해서 재기")대로 돌려봤더니:

```
$ node packages/cli/dist/index.js verify <fixture> --ext tsx
verify FAIL — 3 violation(s) in 4 file(s)
```

`packages/cli/src/verify.ts` + `askewly-design verify <dir>` 가 **이미 존재하고 동작한다.** hex 리터럴과 `rgb/hsl/oklch()` 를 잡고, 토큰 파일(`tokens.css`·`askewly.css`)과 빌드 디렉터리를 제외하고, 위반 시 exit 1을 낸다. 브리프 §5 "Step 1 — 색상 하드코딩 검사기 하나"가 **요구한 것이 거의 그대로 구현돼 있었다.** 브리프는 `scripts/` 20개만 훑고 `packages/cli`를 안 봤다.

그래서 이 horizon은 **"검사기를 만든다"가 아니라 "있는 검사기가 실제로 쓸 만해지고, 실제로 불린다"** 가 된다.

### 리서치가 이미 바꾼 것

**"경고냐 차단이냐"는 이분법이 아니었다.** Kraft는 셋째 답을 쓴다 — **에이전트에게 되먹임**. 사람에게 보고를 첨부하는 것도, 완료를 막는 것도 아니고, 위반을 에이전트가 받아 스스로 고친 뒤 다시 잰다. 사용자가 고른 "경고로 시작"은 이 방향과 어긋나지 않지만, 목적지는 "보고 첨부"보다 **"자가 수정 루프"** 로 잡는 것이 근거에 맞다. DOG6이 이 형태로 배선한다.

## 닫는 기준

각 기준은 close 시 `선언 / 실측 / 판정` 한 줄로 대조한다.

1. **오탐 0** — SVG 내부·주석 안의 색 리터럴을 위반으로 보고하지 않는다. 관측: 이번 실사에서 잡힌 그 fixture 4종이 정확히 뒤집힌다(현재 `svg.tsx`·`comment.tsx` FAIL → PASS).
2. **누락 0** — 한 줄에 hex와 `rgb()`가 같이 있으면 **2건 다** 보고한다. 관측: `oneline.tsx` 위반 수 1 → 2.
3. **실배포** — `npx @askewly/design@<ver> verify <dir>` 가 **이 레포 밖 임시 디렉터리에서** 동작한다. 관측: E404가 아닌 실제 버전 응답 + 레포 밖 실행 exit code.
4. **타이포 계수** — 화면의 font-size 고유값 수를 세고 임계 초과 시 보고한다. 관측: 3단계 fixture PASS / 6단계 fixture FAIL.
5. **매체 근거 등재** — 인쇄·슬라이드 규격이 `llms.txt` 경로에서 fetch된다. 관측: 실제 배포본 fetch 응답(로컬 파일 존재가 아니라 **배포본**) — VL2에서 세운 규율.
6. **매체별 분기** — 마무리 절차가 화면/인쇄/슬라이드에서 서로 다른 게이트를 지시하고, 지면 산출물에 화면용 토큰을 그대로 깔지 않는다. 관측: 세 매체 각 1회 실행, 지시된 게이트가 실제로 달라지는지.
7. **자가 수정 루프** — 위반이 나오면 에이전트가 그 목록을 받아 고친 뒤 다시 잰다. 관측: 의도적 위반 fixture 1회 통과 실행에서 재측정 발생 + 최종 0.
8. **사람 관측** — 사용자가 실제 디자인 작업 1건에서 새 마무리 절차를 겪고, **오탐이 성가신지**와 **차단 승격 가부**를 답한다. 관측: 발화 인용. → 아래 "관측 설계" 참조.
9. **회귀** — 매 milestone `npm run verify` + `npm run typecheck` + 전체 테스트 PASS.

### 관측 설계 — 사람만 답할 수 있는가 (직전 2 horizon의 교훈 적용)

`editor-color-and-token-editing`과 `vocabulary-in-use`가 **연속으로 사람 관측 게이트에서 미달·부분**으로 닫혔다. 원인은 같았다 — 기계가 답할 수 있는 것을 사람에게 물었다. 그래서 기준 8의 과업은 **사람만 답할 수 있는 것으로만** 구성한다.

| 물음 | 사람만 답할 수 있나 | 판정 |
|---|---|---|
| 검사기가 위반을 잡나 | ❌ 기계가 잰다 | 관측 대상 아님 → 기준 1·2·4 |
| **오탐이 성가신가** | ✅ 성가심은 사람만 안다 | **관측 대상** |
| **차단으로 올려도 되겠나** | ✅ 규율/마찰 trade-off는 사용자 소유 | **관측 대상** |
| 배포된 패키지가 도나 | ❌ 실행하면 나온다 | 관측 대상 아님 → 기준 3 |
| **매체별 절차가 자연스러운가** | ✅ 절차 체감은 사람만 안다 | **관측 대상** |

기준 8은 **별도 세션을 잡지 않는다.** 사용자가 어차피 할 디자인 작업 1건에 새 절차를 물려서 겪게 한다 — ECT5에서 재관측이 거절된 이유가 "별도 관측 세션"의 비용이었다.

## 미리 쓰는 실패 회고

이 horizon이 끝났는데 실패했다. 왜 실패했나?

1. **또 만들어놓고 안 불렀다.** 검사기가 좋아졌는데 마무리 절차가 여전히 안 부른다 — 지금 상태 그대로다.
   → 예방: DOG6을 별도 milestone으로 세우고, 기준 7을 "실행에서 재측정이 실제로 발생"으로 잰다. 문서에 한 줄 적는 것으로 닫지 않는다.
2. **오탐이 쌓여 검사기가 무시당했다.** SVG·주석 말고도 오탐 원천이 더 있는데 fixture 4개만 보고 닫았다.
   → 예방: DOG7을 실제 세션에 물려 **사용자가 성가시다고 말하는지**를 잰다. 기준 1은 하한이지 충분조건이 아니다.
3. **npm에 올리는 순간 되돌릴 수 없었다.** 이름·버전·license를 잘못 박고 공개했다.
   → 예방: DOG2의 첫 leaf를 **배포 전 결정 매듭**으로 잡는다(license·scope 공개 여부·버전 정책). `UNLICENSED`인 채로 공개 레지스트리에 올리지 않는다.
4. **슬라이드를 화면처럼 다뤘다.** 16:9 규격만 넣고 정작 "뒷자리에서 안 읽히는 8pt 본문"은 안 잡는다.
   → 예방: DOG5의 게이트는 **기계 검증 가능성 판정을 통과한 항목만** 넣는다. 리서치 §6이 그 판정표다.
5. **화면용 토큰을 또 지면에 깔았다.** 어록 PDF 사고의 재발.
   → 예방: 기준 6에 이 항목을 명시하고, 세 매체 각 1회 실행으로 잰다.
6. **브리프를 그대로 믿고 이미 있는 걸 다시 만들었다.** — 이미 한 번 날 뻔했다. 이 horizon의 첫 실사가 그걸 막았다.
   → 예방: 각 milestone 첫 leaf는 **실행 관측**으로 시작한다. 읽어서 내린 판단은 실사가 아니라 가설이다.
7. **사람 관측이 또 샜다.** 3연속.
   → 예방: 위 "관측 설계" 표를 기준 8의 과업 설계에 그대로 쓴다. 기계가 답할 수 있는 것은 과업에 넣지 않는다.

## 비목표

- **anti-patterns 12클러스터를 코드 검사기로 옮기지 않는다.** "금액 정직성"·"파괴적 동작 보호"는 의미 판단이다. 정규식으로 재려 하면 오탐만 쌓이고 원본 체크리스트의 신뢰가 깎인다(브리프 §7).
- **Kraft 채점기 7개를 다 이식하지 않는다.** `layout-structure`·`icon-usage`는 우리 스택에 해당 없음, LLM 채점기 4개는 anti-patterns가 이미 더 구체적으로 담당(브리프 §3).
- **`spacing-rules`·`animation-stability`** — 규칙부터 정해야 하는 항목. 이번 범위 밖.
- **스킬 본문에 규칙 복제 금지** — `SKILL.md`는 얇은 라우터이고 정본은 원격이다. 검사기 규칙도 원격 문서에 둔다.
- **전역 `CLAUDE.md` 절 재투입** — 사용자 판단으로 제거된 상태다. 이 horizon이 닫히는 것이 재투입 **조건**이지, 이 horizon의 **작업**이 아니다.
- 슬라이드 **제작** 파이프라인(템플릿·렌더러) — 이번엔 규격과 게이트까지다.
