- **무감독 분량: 승인 후 최소 3 무감독 세션.** 5 milestone, 예상 14 leaf.

## 왜 지금 — 직전 horizon이 실측으로 드러냈다

`template-production-hardening`은 닫는 기준 9항 중 8항을 통과했다. 파이프라인은 실제로 돈다:
명함 의뢰 1건이 청사진 선택 → 라이브 소재 생성 → 편집 → 인쇄 발주 가능한 SVG 한 장까지 갔다.

**미달한 하나가 편집 표면이다.** 자동 검증 98 PASS, 브라우저 실조작 통과, 그런데 사람이 화면을
읽지 못했다. 이건 구현 실패가 아니라 **판단 가능성의 부재**다. 검증이 통과해도 사람이 못 읽으면
실사용감은 미확인이고, 그 상태로는 이 편집기를 누구에게도 권할 수 없다.

## 리서치 입력 (근거)

- `research/2026-07-20-editor-ui-horizon-canvas-interaction.md` — Figma·Framer·Penpot 캔버스 UI. 출처 URL·접근일 포함, **확인 실패 14건을 명시적으로 나열**했다.
- `research/2026-07-20-editor-ui-horizon-figma-motion.md` — Figma Motion(Config 2026, Open beta) 실재 확인. 타임라인이 캔버스 안에 있고 컴포넌트 단위로 전파된다.

리서치가 이미 바꾼 것:

1. **Framer는 이번 스펙 소스에서 제외한다.** 공식 문서가 저수준 조작 스펙(핸들 개수·스냅 판정 규칙)을 텍스트로 노출하지 않아 14건 중 5건이 Framer 항목이었다. 문서 리서치로는 더 못 얻는다 — 알고 싶으면 실물 관찰이 별도로 필요하다.
2. **"Figma만큼"은 수치가 아니라 구조로 번역된다.** 회전 핫존 반경 px 같은 값은 공식 출처가 없다(확인 실패 1·2·3번). 우리가 이식할 수 있는 건 **역할 분리 규칙**이다: 변 핸들=단축 리사이즈 / 모서리 핸들=양축 / 회전=모서리 바깥 별도 핫존.
3. **스냅과 측정은 별개 상호작용이다.** 드래그 중 자동 스냅과, 정적 상태에서 모디파이어+호버로 하는 거리 측정을 하나로 뭉치면 둘 다 어설퍼진다.
4. **인스펙터 순서는 Figma·Penpot이 일치한다** — 기하(위치/크기) → 구조(레이아웃/컴포넌트) → 시각(텍스트/Fill/Stroke/Effects) → 내보내기. 우리 인스펙터가 이 순서를 안 지키면 재배치가 우선이다.
5. **Figma Motion은 이번 범위 밖 후보로 둔다.** 조작감·판독성이 안 된 상태에서 모션 타임라인을 얹으면 판단 불가가 한 겹 더 쌓인다. 다만 "모션이 캔버스 안에 살고 컴포넌트로 전파된다"는 구조는 다음 horizon 후보로 `CANDIDATES.md`에 적재한다.

## 닫는 기준 (초안 — 승인 시 확정)

각 기준은 close 시 `선언 / 실측 / 판정` 한 줄로 대조한다.

1. **핸들 역할 분리** — 변·모서리·회전이 서로 다른 조작이고, 시각적으로도 구분된다. 관측: 각 조작의 입력→결과 테스트 + 스크린샷.
2. **스냅과 측정의 분리** — 드래그 중 스냅과 정적 거리 측정이 각각 독립적으로 동작한다. 관측: 두 경로 각각의 실측.
3. **측정 기준 명문화** — 모든 정렬·측정이 바운딩 박스 기준이며 그 정의가 문서에 있다. 관측: 문서 + 좌표 모델 대조 테스트.
4. **레이어 판독성** — 타입이 아이콘으로 구분되고, 선택 시 부모 경로가 펼쳐지며, 검색이 동작한다. 관측: 브라우저 실조작.
5. **인스펙터 순서** — 기하→구조→시각→내보내기 순서를 따르고, 선택 종류에 따라 노출이 달라진다. 관측: 선택 종류별 스크린샷 대조.
6. **판단 가능성** — 사람이 **사전 설명 없이** 화면만 보고 과업 3건(요소 선택·크기 변경·색 토큰 변경)을 수행한다. 관측: 과업별 성공/실패와 막힌 지점. **이 기준이 이 horizon의 존재 이유다.**

## 미리 쓰는 실패 회고

이 horizon이 끝났는데 실패했다. 왜 실패했나?

1. **또 "통과했는데 못 읽는" 상태로 끝났다.** 테스트가 다 초록인데 사람은 여전히 화면을 못 읽는다.
   → 예방: 닫는 기준 6을 **과업 수행 관측**으로 정의한다. 스크린샷을 보여주고 "괜찮나요?"라고 묻는 방식은 직전 horizon에서 이미 실패했다 — 판단할 수 있는 사람에게만 통하는 질문이었다.
2. **게이트가 자기 출력만 봤다.** 직전 horizon에서 **3번** 나온 유형이다(TH9 넘침, TH11 규격 추론, TH11 도련). 렌더 결과를 렌더 코드로 검사하면 통과할 수밖에 없다.
   → 예방: UI 검증은 **입력(사용자 조작)에서 기대값을 유도**하고, 가능하면 우리 코드를 모르는 파서/브라우저 API로 관측한다. probe 없는 게이트는 미완료로 본다.
3. **Figma 스크린샷을 흉내내 픽셀만 닮았다.** 핸들 8개를 그렸지만 변/모서리 역할이 같고, 스냅이 스냅처럼 보이기만 한다.
   → 예방: 각 milestone의 DoD를 "보인다"가 아니라 **"조작하면 이 결과가 나온다"**로 적는다.
4. **수치를 지어냈다.** 회전 핫존 반경, 스냅 허용 오차 같은 값에 출처가 없는데 근거 있는 척 코드에 박았다.
   → 예방: 리서치의 확인 실패 14건 목록을 그대로 들고 간다. 출처 없는 값은 **유추임을 코드에 명시**한다(TH11에서 확립한 규약).
5. **범위가 모션까지 번졌다.** Figma Motion이 매력적이라 타임라인을 얹었고, 정작 선택·레이어는 여전히 안 읽힌다.
   → 예방: 모션은 이 horizon의 **비목표**로 못박고 `CANDIDATES.md`에 둔다.
6. **편집기를 고치느라 제작 파이프라인이 썩었다.** 직전 horizon의 게이트가 방치돼 회귀가 쌓였다.
   → 예방: 매 milestone에서 `npm run verify` + 전체 테스트를 회귀 게이트로 계속 돌린다.
