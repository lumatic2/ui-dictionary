# HORIZON — 편집기에서 색이 색으로 보이고, 바꿔진다

> 생성: 2026-07-21 · Status: **approved 2026-07-21** — 사용자가 5 milestone 묶음과 위임 범위 A(horizon 전체 연쇄)를 승인
> 진행 상태의 정본은 `ROADMAP.md` marker다. 직전 horizon `editor-legibility`의 닫는 기준 6(판단 가능성) 미달에서 개설한다.

## 목표

- AskewlyDesign 편집기에서 **사람이 색을 찾아서 바꿀 수 있게** 만든다.
- 직전 horizon의 마지막 관측에서 사용자가 남긴 문장이 이 horizon의 전부다:
  *"요소 고르고 크기 바꾸는건 알겠는데 색은 어떻게 바꾸는지 모르겠네"*
- 축은 하나가 아니라 넷이다 — **인스펙터 UI · 검증 계층 · 렌더러 · 토큰 조회 API**. 견본 하나 붙이는 일이 아니다(아래 "왜 4표면인가").
- **무감독 분량: 승인 후 최소 3 무감독 세션.** 5 milestone, 예상 14 leaf.

## 왜 지금 — 실측이 정확히 여기를 짚었다

`editor-legibility`는 닫는 기준 6항 중 5항을 통과했다. 선택·핸들·스냅·측정·레이어·인스펙터가 전부 올라갔다.

**미달한 하나가 색이다.** 그리고 이 결함은 **probe 11건과 브라우저 계측이 전부 못 잡았다.**
테스트는 다 초록이었고 계측도 "시각 섹션에 입력이 있다"까지만 말했다. 사람이 한 번 만져보니 5초 만에 나왔다.

관측 직후 같은 화면을 계측한 수치(`evidence/editor-legibility/eu5-judgeability.md`):

| 계측 | 값 |
|---|---|
| 화면 전체에 "색"이라는 단어 | **0건** |
| 색 견본(swatch·color input) | **0개** |
| 색을 바꾸는 유일한 경로 | `Token · fill` — **자유 텍스트 입력** |

즉 색을 바꾸려면 사용자가 ① `Token · fill`이 색이라는 걸 알고 ② 유효한 토큰 이름을 미리 알고 ③ 오타 없이 타이핑해야 한다. 목록도 견본도 자동완성도 없다.

## 리서치 입력 (근거)

- `research/2026-07-21-editor-color-and-token-editing-reference.md` — Figma Variables·Penpot·Framer·Webflow·Tokens Studio 5개 시스템 비교. 출처 URL·접근일 포함, **확인 실패 항목을 대상별로 명시**했다(문서 텍스트로 확인 불가한 시각 세부).
- 레포 실사 2026-07-21 (이 문서 "현 상태 실사" 절) — 코드 근거는 `path:line`으로 남겼다.

리서치가 이미 바꾼 것:

1. **5개 시스템 전부가 "스와치 모양 자체로 바인딩 종류를 말한다"** (Figma: 사각형=변수/원형=스타일, Penpot: 원형=에셋/둥근사각형=토큰). 우리는 스와치가 아예 없다 — 이 축이 통째로 비어 있다.
2. **5개 시스템 전부가 "미바인딩 상태에서 새로 묶는 명시적 어포던스"를 갖는다** (Figma "Apply styles and variables" 버튼, Penpot 우클릭, Framer "+", Webflow 보라 점, Tokens Studio 우클릭). 우리는 이 경로가 없다. 5/5 대비 0 — 이건 취향 차이가 아니라 결손이다.
3. **원시값과 토큰의 공존 모델이 둘로 갈린다** — Figma는 "Detach"로 분리, Penpot은 한 팝오버 안 탭 전환. **택일 대상이었고 사용자가 Figma식을 골랐다**(결정 로그 1).
4. **Penpot·Webflow는 이름 표기법을 그룹 UI로 자동 변환한다**(점표기 / 슬래시). 우리 토큰도 이미 점표기(`surface.canvas`)라 같은 변환이 가능하다.
5. **"묶여있음" 신호로 Webflow는 색(보라 하이라이트), Figma·Penpot은 스와치 모양+라벨을 쓴다.** 색 신호는 라이트/다크에서 대비 문제가 생긴다 — 우리는 형태+라벨을 쓴다.

## 현 상태 실사 (계획 전 확인, 2026-07-21)

**좋은 소식 — 기반은 이미 있다.**

- 토큰 → 실제 색으로 푸는 `documentTokens(setId).resolve()`가 있고, **렌더러가 칠할 때 쓰는 바로 그 함수**다 (`apps/agent-design/src/documentTokens.ts:32`, `apps/agent-design/src/CanvasSurface.tsx:52-91`). 견본에 칠할 색을 새로 구할 필요가 없다.
- 토큰 세트가 평범한 객체라 **목록을 뽑는 건 `Object.keys` 한 줄**이다 (`packages/template-core/src/tokens.ts:21-25`, `apps/agent-design/src/editorTokens.ts:9`). "유효 토큰 목록이 없다"는 데이터가 없어서가 아니라 **UI가 안 물어봐서** 생긴 결함이다.

**나쁜 소식 — EU5 evidence가 본 것보다 깊다.**

1. **검증이 형태만 본다.** `validateNodePropertyEdit`의 token 분기는 값이 `x.y` 꼴인지만 보고 **그 토큰이 실재하는지는 안 본다** (`packages/canvas-core/src/properties.ts:24,46-47`). 그래서 오타는 조용히 **저장되고** 렌더에서만 `data-token-unresolved`로 남는다. EU5 finding 4는 "오타가 조용히 거부된다"였지만 정확히는 **조용히 수용된다** — 더 나쁘다.
2. **이미지 노드는 인스펙터만 고쳐선 안 된다.** 렌더러가 image kind에서 `tokenBindings`를 **아예 참조하지 않는다** (`CanvasSurface.tsx:162-168`). 바인딩 경로만 붙이면 "묶이긴 하는데 안 칠해지는" 상태가 된다.
3. **어휘가 둘인데 메타데이터가 비대칭이다.** 템플릿 세트는 토큰마다 `kind`(color/fontFamily)를 갖는데(`template-core/src/tokens.ts:16-19`), 편집기 세트는 `Record<string,string>` 평면이라 **kind가 없다**(`editorTokens.ts:9`). "색 토큰만 골라 보여준다"를 두 어휘에 똑같이 하려면 생성기(`apps/agent-design/scripts/generate-editor-tokens.mjs`)부터 손대야 한다.
4. **두 어휘를 섞으면 안 된다는 건 이미 코드에 규약으로 박혀 있다** (`documentTokens.ts:12-16` — "템플릿을 편집기 chrome 색으로 칠하면 디자인이 아니라 편집기가 보인다"). 색 선택기는 **문서 자기 세트만** 보여줘야 한다. 이건 이번에 발명할 제약이 아니라 이미 내려진 결정이다.
5. **두 경로의 계약이 어긋나 있다.** `template-core`의 `EXPECTED_KIND`는 `background/fill/color/fontFamily`를 명시 제한하는데(`tokens.ts:74-80`), canvas 렌더 경로는 별도로 키를 하드코드한다(`CanvasSurface.tsx:59-69`). 같은 개념이 두 군데 따로 산다.

### 왜 4표면인가

EU5 finding 4건은 **증상**이고, 그 증상들이 각각 다른 층에 뿌리를 둔다:

| EU5 finding | 실제 뿌리 | 표면 |
|---|---|---|
| 색을 색으로 안 보여준다 | 인스펙터가 자유 텍스트 입력 | 인스펙터 UI |
| 유효 토큰 목록·자동완성 없음 | 토큰 **열거 API가 없다** + 검증이 실재를 안 본다 | 조회 API · 검증 계층 |
| 토큰을 새로 묶는 경로 없음 | 미바인딩 어포던스 부재 | 인스펙터 UI |
| (이미지 노드) 색 변경 불가 | **렌더러가 참조조차 안 한다** | 렌더러 |

## 닫는 기준

각 기준은 close 시 `선언 / 실측 / 판정` 한 줄로 대조한다.

1. **토큰 실재 검증** — 존재하지 않는 토큰 이름은 **저장되지 않고** 사유가 화면에 뜬다. 관측: 오타 입력 probe(거부 + 사유 문자열), 신규 `data-token-unresolved` 발생 0건.
2. **색 견본** — 색 컨트롤이 해석된 실제 색을 보여준다. 관측: **EU5 계측의 정확한 역** — 화면에 "색" 단어 ≥1건, 견본 ≥1개(EU5는 0건/0개였다).
3. **어휘 격리** — 선택기에 문서 자기 세트 토큰만 나온다. 관측: 템플릿 문서에서 편집기 토큰 노출 0건, 편집기 문서에서 템플릿 토큰 0건 — **양방향**.
4. **묶기·풀기 왕복** — 미바인딩 → 바인딩 → detach → 원시값 → 재바인딩이 전부 동작하고 undo로 되돌아온다. 관측: 오퍼레이션 테스트 + 역함수 테스트 + 브라우저 실조작.
5. **이미지 노드 실칠** — 이미지 노드의 색 바인딩이 렌더에 실제로 반영된다. 관측: 계산된 스타일 값 대조(렌더 코드가 아니라 브라우저 계산값).
6. **판단 가능성** — 사람이 **사전 설명 없이** 화면만 보고 과업 3건(① 색 토큰 변경 ② 미바인딩 노드에 색 묶기 ③ 원시 색으로 벗어나기)을 수행한다. 관측: 과업별 성공/실패와 막힌 지점. **이 기준이 이 horizon의 존재 이유다.**
7. **회귀** — 매 milestone에서 `npm run verify` + `npm run typecheck` + 전체 테스트 PASS. 관측: 커맨드 exit code.

## 미리 쓰는 실패 회고

이 horizon이 끝났는데 실패했다. 왜 실패했나?

1. **또 "있다"만 만들고 "쓸 수 있다"는 안 물었다.** 견본은 붙었는데 사람은 여전히 색을 못 찾는다.
   → 예방: 닫는 기준 2를 **EU5 계측의 정확한 역**으로 정의한다(0건→≥1). 기준 6은 사람 관측이고, 그게 이 horizon의 존재 이유다.
2. **detach가 토큰 이탈의 문이 됐다.** 원시 hex가 편한 기본 습관이 되어 디자인 시스템 논지가 무너진다. 사용자가 "탈출구 있음"을 골랐으므로 이 리스크는 **실재한다**.
   → 예방: detach 상태를 화면에 명시 표시하고, **detach된 노드 수를 evidence에 계측으로 남긴다.** 조용히 늘어나지 못하게 한다.
3. **두 어휘를 섞었다.** 편집기 chrome 색이 템플릿 팔레트 목록에 나온다 — 이미 한 번 난 사고다(`documentTokens.ts:71-73`의 드롭다운 사고 기록).
   → 예방: 기준 3을 **양방향 0건**으로 잰다. 한쪽만 보는 게이트는 EU2에서 이미 새어 본 변종이다.
4. **렌더러만 고치고 계약 불일치는 그대로 뒀다.** `EXPECTED_KIND`와 canvas 키 목록이 계속 따로 산다.
   → 예방: ECT4 DoD에 두 경로 키 계약 일치 검증을 넣는다.
5. **타입체크를 또 안 돌렸다.** 직전 horizon에서 EU1의 타입 오류 2건이 EU3까지 살아 있었다 — 원인은 "게이트는 있는데 안 돌림"이었다.
   → 예방: 기준 7에 `npm run typecheck`를 명시하고 매 milestone 돌린다.
6. **팝오버가 접근성 함정이 됐다.** 키보드로 못 닫고 포커스가 갇힌다 — shortcuts dialog에서 이미 유사 항목이 유지보수 후보로 남아 있다.
   → 예방: ECT2 DoD에 키보드 조작(열기·이동·선택·Esc 닫기)을 포함한다.
7. **범위가 토큰 편집기로 번졌다.** 토큰 정의를 만들고 고치는 화면을 짓다가 정작 색은 여전히 못 바꾼다.
   → 예방: 결정 2로 **바인딩까지만**을 못박았다. 토큰 정의 생성은 비목표다.

## 비목표

- **토큰 정의 생성·편집** — SSOT(`tokens/askewly.tokens.json`) 쓰기 + 생성기 재실행 + 두 어휘 정책이 한꺼번에 걸린다. 별도 결정 사안(결정 2).
- 다중선택 공통 값 편집 (EU4 이월 finding).
- 측정 판독의 두 요소 연결선 (EU2 이월), 레이어 아이콘 글리프 강화 (EU3 이월).
- stroke·effect — 문서 모델에 없다. 없는 것을 인스펙터에 만들지 않는다(EU4에서 세운 규율 유지).
- 모션·타임라인, 협업, 새 캔버스 엔진(ADR 0006 유지).
- 제작 파이프라인 기능 추가 — 유지·회귀 방지만.

## 결정 로그 (horizon 수준, 2026-07-21 사용자 확정)

- status: resolved
1. **원시 색: 탈출구 있음 (Figma식 detach).** 기본은 토큰이고, 명시적 detach로 벗어나면 리터럴을 쓸 수 있다. 벗어난 상태는 화면에 표시된다. → 프리모템 2가 이 결정의 리스크를 맡는다.
2. **범위: 바인딩까지만.** 기존 토큰을 고르고·묶고·푸는 데까지. 토큰 정의 생성은 비목표.
3. **이미지 노드: 렌더러까지 확장.** 인스펙터만 고쳐 "묶이는데 안 칠해지는" 상태를 만들지 않는다.
- 추가 사용자 결정 예상: ECT5 관측 대상 문서·과업 문구(그 시점에 확인).
