# HORIZON — 용어가 실제로 쓰이는 흐름 (vocabulary-in-use)

> 생성: 2026-07-21 · ROADMAP marker: `harness:goal id="vocabulary-in-use"` · 상태: active
> Supersedes: `element-decision-layer` 설계 번들(2026-07-21 개설 당일 확장 — 흐름 실사가 갭을 다시 씀)
> 위계: Objective(`OBJECTIVE.md`) → **Horizon**(이 문서) → Milestone(`plans/<date>-<id>-<slug>.md`) → Step.

## 목표

`askewly-design`이 호출됐을 때 에이전트가 지나는 실제 경로에 **UI 용어 사전을 집어넣고**, 요구 한 문장에서 요소를 판정해 구현 자산까지 착지하는 흐름을 완성한다.

- **무감독 분량: 승인 후 최소 5 무감독 세션** (직전 horizon 실측 3세션·13 changeset. 이번은 8 milestone·22 leaf로 그릇을 키웠다.)

## 왜 지금 — 실사 계측 (2026-07-21)

| 계측 | 값 |
|---|---|
| `terms.yml` 용어 수 / 파일 크기 | 562개 / 680KB |
| `llms.txt`에 실린 용어 데이터 | **없음** — `groups.yml`(그룹명 57개)만 실림 |
| `entry-protocol.md`의 용어 언급 횟수 | **0회** |
| 레시피가 참조하는 용어 | 81개 = 전체의 **14.4%** (481개는 구현 자산 없음) |
| 배포본에서 `term_refs` 해소 가능 여부 | **불가** — 레시피 45개의 81개 참조가 배포 경로에서 전부 끊김 |

1. **사전이 에이전트 경로에 없다.** 레포 이름이 `ui-dictionary`인데 에이전트가 실제로 쥐는 어휘는 레시피 45개 + 코드 자산 27개다. 562개 사전은 사람이 보는 사이트 데이터로만 존재한다.
2. **끊긴 참조가 배포된 계약 안에 있다.** 레시피 frontmatter의 `term_refs`는 로컬에서는 해소되지만, `ui.askewly.com`에서 프로토콜대로 fetch하는 에이전트에게는 해소 불가능한 이름 81개다.
3. **판정할 대상이 없으면 결정 계층도 없다.** 후보 사이를 가르는 축을 만들어도 고를 수 있는 것이 46개뿐이면 판정의 의미가 얇다. 어휘 배포가 결정 계층보다 앞선다.
4. **선례가 비어 있다.** 컴포넌트 선택을 기계 판독 데이터로 표현한 표준은 존재하지 않는다(DTCG는 토큰 값 교환만). 근거: `research/2026-07-21-element-decision-layer-selection-guidance-prior-art.md`
5. **직전 horizon의 교훈 둘.** ① 선언한 Gap 3개 중 2개가 실제와 달랐다(EU3) — 그래서 이번엔 **실사 milestone(VL1)을 먼저** 둔다. ② probe 11건과 브라우저 계측이 못 잡은 결함을 사람이 한 번 만져서 잡았다 — 그래서 **사람 관측(VL8)으로 닫는다.**

## 범위 밖 (non-goals)

- 공개 사이트의 "이럴 땐 뭘 쓰나" 비교 페이지 — 데이터 정본화 후 별도 horizon.
- AskewlyDesign 앱 캔버스에서의 요소 제안 UX — 후속.
- 481개 미커버 용어 전부에 레시피/코드 자산 만들기 — 이 horizon은 **참조가 해소되고 폴백 경로가 있는 상태**까지다. 자산 제작은 별건.
- 562개 전부에 판별 축 — 헷갈리는 군집 10~15개로 한정(사용자 확정).

## 닫는 기준 (선언 대 실측)

1. **사전이 배포 경로에 있다** — 관측: 생성된 `llms.txt`에 용어 자산 섹션 존재 + 임의 용어 3개를 조회 규약대로 따라가 각각 3 fetch 이내에 항목 본문 도달.
2. **끊긴 참조 0** — 관측: 무결성 검사로 배포본 기준 해소 불가 `term_refs` 0건 (현재 81건).
3. **자산 없는 용어에 경로가 있다** — 관측: 레시피·코드 자산이 없는 용어를 골랐을 때 프로토콜이 지시하는 폴백 경로가 문서에 있고, 그 경로로 실제 컴포넌트 1건을 구현한 증거.
4. **판별 데이터가 기계 검증된다** — 관측: `python scripts/validate-decisions.py` exit 0, 군집 ≥10, 축마다 출처·confidence 존재, 교차 모순 0.
5. **판정이 갈린다** — 관측: VL1 기준선 대비 VL7 처치 정답률의 군집별 향상 폭 수치. 회귀 케이스 0이거나 사유 기록.
6. **우회 경로 0** — 관측: `entry-protocol.md`의 A/B/C 어느 경로로 들어와도 요소 미정 상태에서 결정 단계를 건너뛸 수 없음(문서 추적 + 자기 적용 1건).
7. **사람이 한 문장을 던져 구현까지 왕복한다** — 관측: VL8 과업 3건의 성공/실패와 막힌 지점. 미달이면 미달로 닫는다.

## 미리 쓰는 실패 회고 (프리모템)

**시나리오 1 — 680KB를 통째로 던져 컨텍스트를 태운다.** 용어를 배포하겠다며 `terms.yml` 전체를 llms.txt에 링크로 걸고, 에이전트가 그걸 fetch해 컨텍스트의 상당량을 사전으로 채운 뒤 정작 구현할 여력을 잃는다.
→ 예방: 닫는 기준 1을 **"3 fetch 이내 도달"**이라는 조회 비용으로 걸었다. VL2는 통짜 배포가 아니라 인덱스+샤드 설계가 산출물이며, 실패 probe로 "전체 파일을 받아야만 답이 나오는 경로가 남아 있는지"를 검사한다.

**시나리오 2 — 문서만 늘고 행동은 그대로.** 자산 15개를 잘 만들고 프로토콜에 링크 한 줄을 걸었는데, 에이전트는 여전히 분류→레시피로 직행한다.
→ 예방: 닫는 기준 5·6을 *문서 존재*가 아니라 *행동 변화*(정답률 대조, 우회 경로 추적)로 걸었다. VL1이 기준선을 먼저 박아 대조 대상을 만든다.

**시나리오 3 — 용어를 골랐는데 만들 수가 없다.** 판정은 정확해졌는데 481개 용어에 구현 자산이 없어 에이전트가 "accordion을 쓰세요"까지만 하고 멈춘다. 사용자 입장에선 아무것도 안 바뀐 것과 같다.
→ 예방: VL3이 **자산 없을 때의 폴백 규약**(용어의 anatomy·states·anti_use로 직접 구현하는 경로)을 만들고, 닫는 기준 3이 그 경로로 실제 1건을 구현한 증거를 요구한다.

**시나리오 4 — 실사 없이 지어낸 갭을 고친다.** 직전 horizon에서 선언 Gap 3개 중 2개가 실제와 달랐다(이미 존재하는 기능을 만들려 했다). 같은 실수를 반복해 세션을 태운다.
→ 예방: VL1을 **첫 milestone**으로 두어 구현 전에 현행 경로를 계측한다. 이후 milestone의 Gap 서술은 VL1 장부를 근거로만 쓴다.

**시나리오 5 — 축이 안 갈려서 판정이 감으로 돌아간다.**
→ 예방: VL7 분리력 검증을 별도 milestone으로 세우고, 정답 라벨을 우리 규칙이 아닌 외부 출처에서 가져와 순환 논증을 막는다.
